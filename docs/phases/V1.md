# V1 — Production: useful for one searcher

V1 is the first version that ships to production. It runs on the hybrid Cloudflare + Fly.io topology with Neon + R2 + better-auth ([../architecture/deployment.md](../architecture/deployment.md)).

## Goal

A single searcher can sign up, paste a search URL, see deduplicated cross-source listings, like/dislike them, ask a wiki maintainer questions about a deal, and capture pages from a browser extension into their workspace's wiki — all from a hosted web app.

## Scope

### Storage moves off disk behind the same interfaces

- `EvidenceStore` -> `packages/storage-r2` (R2 / S3-compatible).
- `MetadataStore` -> `packages/storage-neon` (Neon Postgres + Prisma v7; HTTP driver from CF, node-postgres from Fly).
- `WikiStore` -> `packages/wiki-r2` in production (markdown on R2, content-addressed snapshots, index in Neon).
- Disk implementations remain available for tests, local dev, and offline replay.

### Web app

- Single workspace, multi-user (workspaces are tenant boundary from day one per [ADR 0012](../decisions/0012-multi-tenancy-workspace-as-tenant.md)).
- Auth via better-auth with social providers (Google, GitHub, email OTP) — **wired locally**; production CF Pages deploy TBD.
- Listing explorer with Postgres FTS when `DATABASE_URL` is set; BizBuySell scrape-from-search server action; projects list — **partial** ([`apps/web/agents.md`](../apps/web/agents.md)).
- Saved searches, like/dislike/save/pass, per-deal wiki page — **TBD**.
- Hosted on **CF Pages** (target).

### Ingestion

- 3+ marketplace adapters (BizBuySell + 2 more — likely BizQuest and BusinessesForSale).
- **Marketplace broker enrichment (Part A):** `Broker` / `BrokerFirm` / `BrokerListing` migrations land; materialize shared broker rows from `extractBrokerLinks`, listing-side `brokerName`, and BizBuySell + DealStream broker-profile parsers; one canary broker per marketplace exercised end-to-end. Spec: [data-model.md](../architecture/data-model.md#broker-materialization-workflow); per-adapter tests in `packages/scraper/adapters/*/agents.md` and `packages/storage-neon/tests/*-broker-materialization.test.ts`.
- **Broker-direct lane (preview, gated, Part B):** enumerate brokers from BizBuySell broker directory + IBBA ([`packages/broker-directory`](../../packages/broker-directory/agents.md)); [`packages/broker-site`](../../packages/broker-site/agents.md) crawler runs only against `CLEARBOLT_BROKER_SITE_ALLOWLIST`. Decision: [ADR 0016](../decisions/0016-broker-direct-ingestion-lane.md). Full franchise + state-DRE rollout: [V1.5.md](V1.5.md).
- Scraper runs on **Fly.io** via [`apps/scraper-service`](../../apps/scraper-service/agents.md) (and queue workers when pg-boss lands) with the full HTTP-first + Playwright fallback wisdom (see [../../packages/scraper/agents.md](../../packages/scraper/agents.md)).
- Per-domain AIMD throttle + WAF heuristics + AIA TLS handling + `needsBrowser` persistence.

### Dedup

- V0's deterministic + lexical pipeline remains the spine.
- Vector contributor added to the compositional `Scorer` per [ADR 0011](../decisions/0011-vector-pgvector-on-neon-v1.md).
- Embeddings stored in pgvector on Neon.
- Sub-threshold `MergeCandidate`s captured in V0 are re-evaluated by V1's vector pass without a full re-scan.
- Optional LLM judge for uncertain pairs surfaced in a human-review queue.

### Per-deal wiki (Karpathy pattern)

- Wiki maintainer agent ([../../packages/wiki/agents.md](../../packages/wiki/agents.md)) runs on **Fly.io**.
- `wiki-ingest`, `wiki-query`, `wiki-lint` skills.
- Per-workspace, per-deal directory tree on R2 (markdown).
- Triggered on every new `SourceRecord`, `WorkspaceCapture`, or `Transcript`.

- Processing pipeline runs on **Fly.io**: HTML -> Markdown via Defuddle, store both raw + MD in R2, AI extraction via `packages/agents`, route to wiki maintainer.
- Browser extension itself lands in V2.

### Telemetry stack (per [ADR 0014](../decisions/0014-telemetry-stack.md))

- **Layer 1 — operational metrics:** `packages/observability` `MetricsSink` wired to a self-hosted **VictoriaMetrics** single-binary on Fly.io with a persistent volume. Each Fly machine exposes `/metrics`; VM scrapes. CF Workers push via a small batched HTTP exporter on the request path. Cluster-vs-managed decision deferred to V1.5.
- **Layer 2 — operational logs + traces:** pino → stdout (Logpush from CF; Fly log shipper) plus an OpenTelemetry SDK + OTLP exporter for spans. Final OTel backend (Loki + Tempo / Honeycomb / Datadog) chosen at V1.5 against ~30 days of signal volume.
- **Layer 3 — product analytics + agent run traces:** new `packages/telemetry` ships its `ProductEvents` contract with a **self-hosted PostHog** on Fly.io as the default backend. Covers product analytics (funnels, cohorts, feature flags, session replay) AND LLM observability (per-call generations, prompt management, evals) in one store. Workspace-private by default; cross-workspace aggregation requires opt-in and a nightly de-identification pipeline (V1.5). Sensitive fields (financial values, contact PII, capture content, wiki body) never leave the workspace.
- **Layer 4 — observability UI (minimal):** `/admin` route in `apps/web` that:
  - Embeds 3-5 PostHog dashboards via signed `<iframe>` (PostHog supports per-user JWT-scoped embeds).
  - Has a "System metrics" panel that proxies VM `/api/v1/query` for ~3 charts (fetch p95 by adapter, dedup decisions/min, harness session cost/min).
  - Links out to the raw PostHog UI for power users.
- **Closed feedback loop:** Layer 3's combined product + agent events let the agent platform team query "users who saw prompt version X had a 3% higher save-rate than version Y" with a single SQL query — the mechanical basis for "improve agentic or otherwise" iteration.

## Topology recap

- **CF Pages**: web app (includes `/admin` minimal observability UI).
- **CF Workers**: `POST /api/captures` and hot read APIs.
- **Fly.io**: scraper (got + Playwright), transcribe (ffmpeg + faster-whisper), agent runners (wiki maintainer, ranker, quality scorer), queue worker (pg-boss), write/mutation API, **VictoriaMetrics** (Layer 1), **self-hosted PostHog** (Layer 3), OTel collector + log/trace backend (Layer 2).
- **Neon**: Postgres + pgvector + pg-boss tables.
- **R2**: blobs + wiki markdown.
- **better-auth**: shared identity.

## Acceptance criteria

- A new user can sign up, create a workspace, paste a BizBuySell search URL, and within minutes see deduplicated, ranked results in the explorer.
- The same listing appearing on BizBuySell and a broker website resolves to one canonical deal with two `SourceRecord`s and a per-field provenance map.
- A user can click "Capture" in the (V1: server-API only) and have a deal page appear in their workspace wiki within seconds.
- A user can ask "what do we know about this deal?" on the deal page and get a synthesised answer from the wiki maintainer with citations to source pages.
- All workspace-scoped queries pass cross-tenant isolation tests (no leak between workspaces).
- AI Gateway dashboard shows model spend per workspace per task; soft caps enforced.
- A workspace `admin` opening `/admin` sees their workspace's PostHog dashboards (events, funnels) and the system-metrics panel for fetch / dedup / harness cost.
- `agent.llm_called` events arriving in PostHog carry a `promptVersion` and join with product events on `userId` so the agent platform team can compute prompt-vs-funnel deltas.

## Validation criteria

V1 acceptance is met when all items in the [Acceptance criteria](#acceptance-criteria) section above pass the corresponding test:

- All criteria above are **integration** or **smoke** coverage.
- Tests live under `apps/web/tests/`, `services/scraper/tests/`, `services/wiki-maintainer/tests/`, and `scripts/telemetry-health.mjs`.
- Cross-tenant isolation has its own test pack: `apps/web/tests/tenant-isolation/*.test.ts`.
- AI cost cap enforcement: `services/api/tests/ai-budget.test.ts`.
- Telemetry health: `scripts/telemetry-health.mjs::posthog_delivery_rate_99pct`, `::vm_scrape_success_995pct`, `::otel_trace_export_99pct`.
- Admin UI workspace scoping: `apps/web/tests/admin-posthog-scoped.test.ts` (cross-workspace event injection must not appear).
- Per-event privacy: `packages/telemetry-posthog/tests/sensitivity-guard.test.ts` and `::deny-list.property.test.ts`.
- Marketplace broker enrichment: `packages/storage-neon/tests/bizbuysell-broker-materialization.test.ts` and at least one other adapter broker-materialization test (BizQuest or DealStream) pass.
- Broker-direct preview (gated): `packages/broker-directory/tests/bbs-dir.test.ts`, `packages/broker-directory/tests/ibba.test.ts`, and `packages/broker-site/tests/allowlist-blocks.test.ts` pass; broker-site golden eval may be advisory until allow-list pilot starts.

## Out of scope (V2)

- The actual browser extension build (V1 ships only the backend capture API; user-side capture happens via curl / API in V1).
- Multi-user collaboration features (multi-user is supported, but features like shared notes, mentions, real-time collab, etc. land in V2).
- Outreach (email/phone) sending.
- Container sandboxes for the agent harness (V1 uses `virtual` in-process Sandbox).
- Provider profiles, deal-team shopping, public API/webhooks.
- **Full federated admin UI** (V2). V1 ships only the minimal `/admin` with PostHog embeds + 3 VM-proxied panels.
- **Cross-workspace de-identification pipeline** (V1.5). V1 keeps everything workspace-scoped.
- **Langfuse** as a second telemetry backend (V2+ only if PostHog LLM observability is found insufficient).
